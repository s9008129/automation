# 內部網路網頁素材蒐集工具使用指南

> 這份手冊是給第一次使用的人。你可以把這個工具想成「幫網站拍照、畫地圖、記操作」，再把這些附件交給 AI，請它幫你寫自動化腳本。

## 1. 為什麼這次改用 Playwright？

如果你原本比較熟 Selenium，可以先把兩者想成兩種找店方式：

- **Selenium**：比較像「拿地址找店」。你知道店大概在哪裡，但只要門口改一下、畫面換一下，有時就要重新確認。
- **Playwright**：比較像「看招牌找店」。它比較容易直接抓到真正要按的按鈕、欄位或文字，所以面對現在常見的網站，通常更穩一點。

這不是說 Selenium 不好。Selenium 很像大家用很久的老工具：成熟、資料多、很多同事也熟。  
但這次我們要做的是 **蒐集網站素材，再交給 AI 產生腳本**。如果說 Selenium 比較像要自己翻出好幾樣工具，Playwright 就比較像一個已經把「錄製、截圖、看畫面結構」放在一起的工具包，所以這條流程通常更順。

| 工具 | 好處 | 可能要留意 |
|---|---|---|
| Selenium | 大家熟、範例多、很多舊系統也找得到做法 | 畫面一改、等待時機一變，有時就要多調整 |
| Playwright | 對現代網站通常比較穩，也更適合錄製、截圖、蒐集快照 | 需要習慣新的做法，舊 Selenium 範例不能直接整段搬過來 |

**一句話記住：** Selenium 像老工具，Playwright 像新工具箱。這次不是要否定 Selenium，而是選一個更適合目前任務的工具。

## 2. 安裝方法

先記住這一句就夠了：

- **Windows 11 一般使用者主流程**：`.\install.ps1` → `.\launch-chrome.ps1` → `.\collect.ps1`
- **技術人員在原始專案補齊環境**：`.\setup.ps1` 或 `npm run setup`

> 如果你拿到的是技術人員準備好的 **完整離線包**，通常不需要自己研究 npm 指令；照著本文的 Windows 主流程做即可。

### 2.1 先分清楚 5 個入口在做什麼

| 指令 / 檔案 | 給誰用 | 它真正做的事 | 重要提醒 |
|---|---|---|---|
| `.\install.ps1` | Windows 一般使用者 | 呼叫 `setup.ps1 -Offline`，**只做離線完整性檢查** | **不是線上下載安裝器**，不會幫你上網抓套件 |
| `.\setup.ps1` | Windows 技術人員 / 原始專案使用者 | 檢查 Node.js、必要時補齊 `node_modules`，並安裝 Playwright Chromium | 真正會補齊依賴的是它 |
| `npm run setup` | macOS / Linux / Windows 都可 | 跨平台入口；會自動轉去 Windows 的 `setup.ps1` 或 macOS / Linux 的 `scripts/setup.sh` | 想記一個共用指令時用這個 |
| `.\launch-chrome.ps1` | Windows 使用者 | 啟動 Chrome Debug 模式，建立獨立的 `chrome-debug-profile/` | 讓蒐集工具能連到你登入中的 Chrome |
| `.\collect.ps1` | Windows 使用者 | 啟動素材蒐集工具 | 蒐集結果會放到 `materials/` |

### 2.2 `install.ps1` 與 `setup.ps1` 的差異

這兩個名稱很像，但角色完全不同：

- **`install.ps1` = 離線完整性檢查入口**
  - 適合一般使用者
  - 會確認你拿到的資料夾是不是「可直接用的完整包」
  - 會寫入 `logs/setup-*.log`
  - **不會**主動下載 `node_modules`
  - **不會**主動下載 Playwright Chromium

- **`setup.ps1` = 真正補齊執行環境的腳本**
  - 適合技術人員，或你現在手上是「原始專案資料夾」而不是完整離線包
  - 需要可用的 **Node.js v20+**
  - 若缺少套件，會自動補齊 `node_modules`
  - 若缺少 Playwright Chromium，會安裝到專案內的 `.playwright-browsers/`

簡單講：

- **一般使用者**：先跑 `.\install.ps1`
- **技術人員 / 準備環境的人**：跑 `.\setup.ps1` 或 `npm run setup`

### 2.3 Windows 11 + PowerShell 7.x 主流程（推薦）

這是給非技術人員的主線流程，照做就可以。

#### 步驟 1：用 PowerShell 7 開啟專案資料夾

請在 **Windows 11 + PowerShell 7.x** 中操作。

如果你想先確認 PowerShell 版本，可輸入：

```powershell
$PSVersionTable.PSVersion
```

看到 `Major` 是 `7` 即可。

#### 步驟 2：先做離線完整性檢查

```powershell
.\install.ps1
```

你可以把這一步想成：「先確認這份工具包有沒有缺件」。

正常情況下，它會檢查：

- 可用的 Node.js runtime（版本要求是 **v20 以上**）
- `node_modules/` 是否齊全
- Playwright Chromium 是否已準備好
- `install.ps1`、`collect.ps1`、`launch-chrome.ps1`、`collect-materials.ts` 等入口檔是否存在

如果你拿到的是完整離線包，這一步應該會通過。

#### 步驟 3：如果你手上是原始專案，才需要真的補安裝

如果 `.\install.ps1` 告訴你安裝包不完整，或你根本不是拿到別人打包好的離線包，而是直接拿到原始專案資料夾，請改用：

```powershell
.\setup.ps1
```

或：

```powershell
npm run setup
```

這一步才是「真的去補齊依賴」。

#### 步驟 4：啟動 Chrome Debug 模式

```powershell
.\launch-chrome.ps1
```

這一步會：

- 建立或重用 `chrome-debug-profile/`
- 開啟專用 Chrome 視窗
- 讓蒐集工具可以透過 `http://localhost:9222/json/version` 連到 Chrome

#### 步驟 5：登入內部網站後開始蒐集

在剛開啟的 Chrome 裡登入你的內部網站，然後在另一個 PowerShell 視窗執行：

```powershell
.\collect.ps1
```

如果你只想先做最小驗證，也可以在目標頁面開好後執行：

```powershell
.\collect.ps1 --snapshot
```

### 2.4 macOS / Linux 相容安裝流程

如果你不是在 Windows，而是在 macOS 或 Linux，請改用 shell 版本：

#### 線上補齊模式

```bash
./scripts/setup.sh
```

或：

```bash
npm run setup
```

#### 離線檢查模式

```bash
./scripts/setup.sh --offline
```

或：

```bash
npm run setup:offline
```

#### 啟動 Chrome Debug

```bash
./scripts/launch-chrome.sh
```

#### 開始蒐集

```bash
npm run collect
```

如果只想快速確認目前頁面可蒐集：

```bash
npm run collect -- --snapshot
```

> macOS / Linux 也需要 **Node.js v20+**。差別只在於入口檔不是 PowerShell，而是 `setup.sh` / `launch-chrome.sh`。

### 2.5 Node.js 版本要求（重要）

本專案要求 **Node.js >= 20**。

- `package.json` 已明確限制 `node: ">=20.0.0"`
- `setup.ps1` 與 `scripts/setup.sh` 也都會檢查主版本是否至少為 20

對非技術人員來說，請這樣理解：

- **你拿到完整 Windows 離線包時**，可能不需要自己另外安裝 Node.js，因為打包者可能已經把可用 runtime 一起放進來
- **但工具實際使用的 Node.js 版本，仍然必須是 v20 以上**
- **如果你是技術人員自己準備環境**，請先確認：

```powershell
node -v
```

或：

```bash
node -v
```

應看到 `v20.x.x`、`v21.x.x`、`v22.x.x` 這類版本。

### 2.6 `npm install` 會裝哪些套件？白話版說明

當技術人員在可連網環境中執行 `npm install`（或由 `setup.ps1` 自動補齊依賴）時，會安裝下列核心套件：

| 套件 | 用途（白話說明） |
|---|---|
| `playwright` | 這是整個工具的核心自動化引擎，負責連到 Chrome Debug、抓 ARIA 快照、截圖、錄製操作流程 |
| `@types/node` | 讓 TypeScript 看得懂 Node.js 內建功能，減少程式碼出錯機會 |
| `tsx` | 讓 `.ts` TypeScript 檔可以直接執行，不用先手動編譯成 `.js` |
| `typescript` | TypeScript 的型別檢查與編譯基礎工具，等於整個 TypeScript 開發環境的底座 |

安裝完成後，這些套件通常會出現在：

```text
node_modules/
```

### 2.7 Playwright Chromium 是什麼？會裝到哪裡？

你可以把 **Playwright Chromium** 想成「Playwright 工具鏈要用的瀏覽器執行元件」。

- 它**不是**你平常日常使用的那個 Google Chrome
- 本專案平常蒐集時，主要會連到你自己開啟的 **Chrome Debug**
- 但安裝流程仍會檢查或準備 Playwright Chromium，確保 Playwright 執行環境完整

常見位置如下：

| 平台 | 常見位置 | 說明 |
|---|---|---|
| Windows（完整離線包推薦） | 專案內 `.playwright-browsers/` | `setup.ps1` 會優先把 Chromium 裝在專案內，方便整包帶去別台電腦 |
| Windows（可能的系統快取） | `%LOCALAPPDATA%\ms-playwright` 或 `%USERPROFILE%\.cache\ms-playwright` | 若你之前在本機裝過 Playwright，可能已存在這些快取 |
| macOS | `~/Library/Caches/ms-playwright` | `scripts/setup.sh` 的預設安裝位置通常在這裡 |
| Linux | `~/.cache/ms-playwright` | 常見的 Playwright 瀏覽器快取位置 |
| 任意平台（有自訂時） | `PLAYWRIGHT_BROWSERS_PATH` 指向的路徑 | 若有設定環境變數，Playwright 會用你指定的位置 |

如果是 **Windows 完整離線包**，最理想的狀態是專案裡直接看得到：

```text
.playwright-browsers/
```

### 2.8 安裝 / 啟動 / 蒐集時，會建立哪些資料夾？

下面這張表可以幫你快速分辨「哪些是正常會長出來的資料夾」。

| 階段 | 會建立或使用的資料夾 / 檔案 | 何時出現 | 用途 |
|---|---|---|---|
| 安裝檢查 / setup | `logs/` | 第一次執行 `install.ps1`、`setup.ps1`、`setup.sh` 時 | 存放安裝日誌，例如 `setup-*.log` |
| 套件安裝 | `node_modules/` | 執行 `setup.ps1`、`npm run setup`、`npm install` 後 | 存放 npm 套件 |
| Playwright 瀏覽器 | `.playwright-browsers/` | Windows 線上補齊 Chromium 後，或完整離線包已預放時 | 存放 Playwright Chromium |
| Chrome 啟動 | `chrome-debug-profile/` | 第一次執行 `.\launch-chrome.ps1` 或 `./scripts/launch-chrome.sh` 時 | 專用 Chrome 設定檔，不影響你平常的 Chrome |
| 蒐集輸出根目錄 | `materials/` | 第一次蒐集時 | 存放所有任務成果 |
| 單次任務成果 | `materials\<時間戳記_任務名稱>\` | 每次蒐集時 | 單一任務的輸出目錄 |
| 任務細項 | `aria-snapshots/`、`screenshots/`、`recordings/`、可選的 `html-sources/` | 建立任務目錄時 | 存放快照、截圖、錄製與 HTML 原始碼 |
| 蒐集日誌 | `logs/collect-materials-*.log` | 每次執行蒐集時 | 記錄蒐集過程與錯誤 |

另外，以下檔案建議在你拿到專案資料夾時就已存在：

- `package.json`
- `install.ps1`
- `setup.ps1`
- `collect.ps1`
- `launch-chrome.ps1`
- `collect-materials.ts`
- `scripts/run-setup.mjs`
- `scripts/setup.sh`

### 2.9 線上模式 vs 離線模式

| 模式 | 常用指令 | 會不會下載東西 | 適合誰 |
|---|---|---|---|
| 線上模式 | `.\setup.ps1`、`npm run setup`、`./scripts/setup.sh` | 會。缺少 `node_modules` 或 Chromium 時，會補齊 | 技術人員、原始專案環境、可連網的準備機 |
| 離線模式 | `.\install.ps1`、`.\setup.ps1 -Offline`、`npm run setup:offline`、`./scripts/setup.sh --offline` | 不會。只檢查現有資源是否完整 | 一般使用者、完整離線包驗收、內網電腦 |

請特別記得：

- **`install.ps1` 屬於離線模式**
- **`install.ps1` 不是線上下載入口**
- **真正會補齊 `node_modules` / Playwright Chromium 的是 `setup.ps1` 或 `npm run setup`**

### 2.10 具體驗證安裝成功的命令

下面提供「可直接照打」的驗證方式。

#### Windows 11 + PowerShell 7.x

1. 驗證 PowerShell 版本：

```powershell
$PSVersionTable.PSVersion
```

2. 如果你是技術人員，驗證 Node.js 版本：

```powershell
node -v
```

3. 驗證離線包完整性：

```powershell
.\install.ps1
```

4. 驗證關鍵目錄是否存在：

```powershell
Test-Path .\logs
Test-Path .\node_modules
Test-Path .\.playwright-browsers
Test-Path "$env:LOCALAPPDATA\ms-playwright"
Test-Path "$env:USERPROFILE\.cache\ms-playwright"
```

一般來說：

- **完整 Windows 離線包**：`.playwright-browsers` 最理想是 `True`
- **若專案內是 `False`，但系統快取是 `True`**：表示本機可能已有 Playwright 快取可用

5. 啟動 Chrome Debug 並驗證：

```powershell
.\launch-chrome.ps1
Invoke-RestMethod http://localhost:9222/json/version
```

只要第二行有回傳 JSON 內容，就代表 Chrome Debug 連線正常。

6. 做一次最小蒐集驗證（先在 Chrome 開好目標頁面）：

```powershell
.\collect.ps1 --snapshot
```

成功後，應可在 `materials/` 下看到新的任務資料夾。

#### macOS / Linux

1. 驗證 Node.js 版本：

```bash
node -v
```

2. 執行安裝 / 補齊：

```bash
./scripts/setup.sh
```

3. 驗證離線檢查：

```bash
./scripts/setup.sh --offline
```

4. 驗證關鍵目錄 / 快取位置：

```bash
test -d node_modules && echo "node_modules OK"
test -d .playwright-browsers && echo ".playwright-browsers OK" || true
test -d "$HOME/Library/Caches/ms-playwright" && echo "macOS cache OK" || true
test -d "$HOME/.cache/ms-playwright" && echo "Linux cache OK" || true
```

5. 驗證 Chrome Debug：

```bash
./scripts/launch-chrome.sh
curl http://localhost:9222/json/version
```

6. 做一次最小蒐集驗證：

```bash
npm run collect -- --snapshot
```

成功後，應可在 `materials/` 下看到新的任務資料夾。

### 2.11 什麼時候需要找技術人員？

出現下列情況，再請技術人員接手即可：

- `.\install.ps1` 明確告訴你「安裝包不完整」
- `setup.ps1` 或 `setup.sh` 補齊失敗
- `node -v` 顯示版本低於 20（技術人員環境）
- `.\launch-chrome.ps1` 啟動後，`http://localhost:9222/json/version` 沒有回應
- `.\collect.ps1 --snapshot` 無法產生 `materials/` 成果

如果你是幫同事準備安裝包的技術人員，請先在可準備環境的電腦執行：

```powershell
.\scripts\prepare-offline-bundle.ps1
```

完成後再把整個資料夾帶到內網給一般使用者。

## 3. 這個工具是做什麼的？

這個工具用來蒐集內部網站的素材，方便之後交給 AI 產生 Playwright 自動化腳本。

- **ARIA 快照**：像網站的「骨架圖」，讓 AI 知道頁面上有哪些按鈕、欄位、表格。
- **截圖**：像「畫面照片」，讓你和 AI 確認畫面長相。
- **錄製檔**：像「操作示範」，讓 AI 知道你先按哪裡、後做什麼。
- **metadata.json**：像「說明卡」，記錄這次蒐集的基本資訊。
- **log**：像「看診紀錄」，出錯時可以交給 AI 協助排查。

## 4. 新手主線：先用互動模式（1）

如果你是第一次用，或只是想穩穩完成一次蒐集，**直接走這條**：

1. 啟動專用 Chrome：

```powershell
.\launch-chrome.ps1
```

2. 在開啟的 Chrome 內登入你的內部網站。
3. 另外開一個 PowerShell 視窗，執行：

```powershell
.\collect.ps1
```

4. 畫面出現選單時，**先選互動模式（1）**。
5. 依照提示選頁面、輸入名稱，工具會幫你存好素材。

> 新手、第一次使用、平常大多數情況：都先用 `.\collect.ps1` → 互動模式（1）。

如果你想把「操作流程」也示範給 AI，把它當成**互動模式裡的延伸**就好：照畫面提示繼續錄製即可；只有想另外補錄操作時，才再用 `.\collect.ps1 --record 錄製名稱`。

## 5. 只有這 2 種例外，才改用其他模式

| 情境 | 建議模式 | 什麼時候用 |
|---|---|---|
| 你已經有固定設定檔，而且要重複批次蒐集 | `.\collect.ps1 --auto`（模式 2 / 自動模式） | 適合例行工作；第一次先用互動模式確認設定沒問題 |
| 你只想快速抓「現在這一頁」 | `.\collect.ps1 --snapshot`（模式 3 / 快照模式） | 適合臨時補一張快照或截圖；不是完整任務的主流程 |

除了這兩種情況，先不要跳去別的模式，直接回到互動模式（1）即可。

## 6. 蒐集完成後，去哪裡看結果？

每次蒐集都會建立一個新的任務資料夾，位置在：

```text
materials\<時間戳記_任務名稱>\
```

常見內容如下：

```text
materials\<時間戳記_任務名稱>\
├─ aria-snapshots\   ← 最重要，AI 主要靠這個理解頁面
├─ screenshots\      ← 畫面截圖
├─ recordings\       ← 操作錄製檔
├─ metadata.json      ← 本次蒐集資訊
└─ summary-report.md  ← 摘要說明
```

如果執行過程有問題，請再看：

```text
logs\
```

裡面最新的 `.log` 檔。

## 7. 什麼情況要蒐集哪些素材？

- **只想讓 AI 看懂頁面**：至少蒐集 **ARIA 快照 + 截圖**。
- **想讓 AI 更容易照著你的步驟做**：再加上 **錄製檔**。
- **想讓 AI 更快理解整體背景**：一起附上 **metadata.json**。
- **腳本失敗、要請 AI 幫你除錯**：再附上 **log**。

不知道怎麼選時，最簡單的做法就是：**把同一次任務資料夾中的檔案，用附件方式一起上傳給 AI。**

## 8. 怎麼把素材交給 AI

1. 找到同一次蒐集的任務資料夾。
2. 在 AI 對話中，用 **附件上傳** 這些檔案：
   - `aria-snapshots\` 裡的檔案
   - `screenshots\` 裡的檔案
   - `recordings\` 裡的檔案（如果有）
   - `metadata.json`
3. 如果是除錯，再加上：
   - 你目前的腳本檔
   - `logs\` 裡相關的 `.log`
4. **不要把長篇 ARIA、錄製檔、metadata 或 log 內容整段貼進訊息裡。** 直接上傳附件就好。

## 9. 生成腳本範本

先把 **ARIA 快照、截圖、錄製檔（如果有）、metadata.json** 用附件方式上傳，再貼上下面這一段：

```text
請根據我剛剛上傳的附件，幫我產生一份可直接執行的 Playwright TypeScript 腳本。我的目標是【請替換：你想自動完成的工作，例如：登入後查詢資料並匯出結果】；成功標準是【請替換：你怎樣算成功，例如：畫面顯示查詢結果並完成下載】；執行環境是【請替換：你的環境，例如：Windows 11、已開啟 Chrome Debug、已登入內部網站】。請優先根據附件中的 ARIA 快照判斷頁面結構，參考錄製檔理解操作順序，用截圖輔助確認畫面，用 metadata.json 補足背景與順序；如果附件不足，請先明確告訴我缺哪個檔案或哪個步驟，不要自行猜測。請輸出：1. 流程摘要，2. 完整 TypeScript 腳本，3. 需要我先準備的環境變數或參數，4. 容易失敗的步驟與替代做法。附件請直接讀取我上傳的 ARIA 快照、錄製檔、截圖、metadata.json，不要要求我把長內容貼到訊息裡。
```

## 10. 除錯範本

先把 **目前腳本、出錯頁面的 ARIA 快照、相關截圖、錄製檔（如果有）、metadata.json、log** 用附件方式上傳，再貼上下面這一段：

```text
請幫我找出這份 Playwright TypeScript 腳本失敗的原因，並直接提供可替換的修正版。我的目標是【請替換：原本想完成的工作】；我預期看到的結果是【請替換：成功時應該發生什麼】；實際遇到的問題是【請替換：錯誤訊息或失敗現象】；大約失敗在【請替換：第幾步或哪個頁面】。請先根據我上傳的附件比對腳本、ARIA 快照、錄製檔、截圖、metadata.json 與 log，告訴我最可能的 1 到 3 個原因，再提供完整修正版或清楚標示要替換的程式碼段落，並說明下次如何避免同樣問題；如果附件還不夠，請直接列出還缺哪個檔案，不要自行猜測。附件請直接讀取我上傳的腳本、ARIA 快照、錄製檔、截圖、metadata.json、log，不要要求我把長內容整段貼到訊息裡。
```

## 11. 安全提醒

- **不要上傳密碼、token、`.env` 檔案。**
- 錄製檔雖然通常會自動清掉密碼，但 **上傳前還是請快速看一眼**。
- 截圖可能拍到個資或敏感資訊，帶出內網前請先確認。
- 不確定要給 AI 哪些檔案時，優先交 **同一次任務資料夾** 的檔案，不要把不同任務混在一起。

## 12. 常見問題 / 快速排除

| 問題 | 你可以先做什麼 |
|---|---|
| Chrome 連不上 | 重新執行 `.\launch-chrome.ps1`，並確認你登入的是那個新開的 Chrome |
| 找不到要蒐集的頁面 | 先把目標網站頁面切到前景，再執行 `.\collect.ps1`，並選互動模式（1） |
| 只想快速抓目前這一頁 | 這時才用 `.\collect.ps1 --snapshot`；其他情況先回 `.\collect.ps1` |
| 錄製完沒有看到檔案 | 再試一次錄製；如果還是不行，先把已有素材交給 AI，再附上 log 請它幫你判斷 |
| AI 說資料不夠 | 補上同一次任務的 ARIA 快照、截圖、錄製檔、metadata.json |
| 腳本執行失敗 | 把腳本、出錯頁面的 ARIA 快照、截圖、相關 log 一起用附件上傳，套用上面的除錯範本 |

## 13. 一句話記住

**先執行 `.\install.ps1`，再開 `.\launch-chrome.ps1`，登入後執行 `.\collect.ps1`，先選互動模式（1）。**
只有兩個例外：**固定設定反覆批次**才用自動模式，**只抓目前這一頁**才用快照模式。
